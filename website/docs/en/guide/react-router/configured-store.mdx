# Bound Store

The default `useSearchValue` and `useSearchValues` hooks can be used directly without manually creating a Provider. They use the default singleton search store and fit most applications with only one React Router runtime.

When an application needs to isolate stores, bind to a specific router runtime, or reduce some unrelated updates, use `@guanriyue/decurl/configured`:

```tsx
import { createReactRouterSearch } from '@guanriyue/decurl/configured';

const search = createReactRouterSearch({
  flushMode: 'debounce',
  flushDelay: 80,
});
```

`createReactRouterSearch` returns APIs bound to the same store:

- `Provider`
- `useSearchValue`
- `useSearchValues`
- `store`

## Basic Usage

```tsx
import { BrowserRouter } from 'react-router';
import { createReactRouterSearch } from '@guanriyue/decurl/configured';

const search = createReactRouterSearch();

export function App() {
  return (
    <BrowserRouter>
      <search.Provider>
        <SearchPanel />
      </search.Provider>
    </BrowserRouter>
  );
}
```

`Provider` connects the current React Router runtime to this bound store. Components inside the Provider should use `search.useSearchValue` and `search.useSearchValues`, and they will share the same optimistic search snapshot.

## Flush Configuration

Business code usually does not need a long `flushDelay`. It is typically used to reduce router navigation frequency caused by high-frequency interactions such as inputs and filters.

```tsx
const search = createReactRouterSearch({
  flushMode: 'debounce',
  flushDelay: 80,
});
```

Common values can start from a few dozen milliseconds and be adjusted by interaction type. Input interactions fit `debounce` better; click interactions can usually use the default strategy or a shorter delay.

If `memo` is not used at all, React Router may still re-render the current docs page component tree after location confirmation, so the render count of unrelated demo components may increase once during the URL confirmation phase. This is not Decurl actively notifying unrelated field changes. It is parent render propagation caused by React Router location updates.

This entry can still reduce some renders, mainly because:

- Bound hooks consume the bound store snapshot and do not directly depend on React Router `useLocation` inside the hook like default hooks do.
- When React Router location is confirmed, bound hooks do not necessarily re-run because they called `useLocation` themselves.

## When to Use It

Prefer the default hooks first. Consider a bound store when:

- The application has multiple React Router runtimes. Default hooks use the default singleton store, and sharing one default store across multiple router instances can make runtime boundaries unclear. In that case, create a bound store for each router instance.
- You need to reduce hook consumer renders caused by React Router location changes. Bound hooks do not directly depend on React Router `useLocation`, so they are less affected by the location confirmation phase.
- You want to configure `flushMode` or `flushDelay` for a group of hooks.
- You need to wire the runtime explicitly through a data router instance.

Bound store is an isolation and optimization entry. It is not a prerequisite for basic hooks.

Note that React Router may still re-render the route tree after location confirmation. A bound store can prevent hook consumers from directly subscribing to `useLocation`, but it cannot prevent React Router parent renders from propagating to child components. For stable heavy child components, combine it with `React.memo` or `useMemo` for further optimization.
